\title{Controlled Generative Development: Structural Realization as a Checkable Layer Between Intent and Code}
\author{Alexander Fedorenko \\ Independent Researcher \\ alex.grig.fed@gmail.com}
\date{}

\begin{document}
\maketitle

\begin{abstract}
This paper proposes a new formulation of the problem for AI-native development in which the central object before code generation is neither the prompt nor the code, but the \textbf{structural realization} of the task. We describe CGD through the architectural framing \textbf{Intent $\to$ Structural Realization $\to$ Execution} and through the top-level pair $\mathbf{D = \langle I, R \rangle}$, where $I$ denotes an intent representation and $R$ denotes a structural realization. For CGD, the formal core of the second element is given by the triple $\mathbf{R = \langle G, \Sigma, C \rangle}$: a typed graph of functions and tables, table signatures, and function contracts. This formulation makes it possible to separate the free interface of intent from a strict layer suitable for verification, analysis, modification, and preparation for transfer into execution backends. The central thesis of the paper is that practical controllability in AI-assisted development begins not with ever more elaborate input prompts, but with fixing the \textbf{model output} through an output contract, abstraction control, and a verification profile. Across three demonstrations, we show that the output contract makes architectural omissions observable, abstraction control reduces drift in the level of description, and verification-driven dialogue makes it possible to bring a structural realization to specification closure before code generation. The paper is presented as conceptual and architectural: we do not claim a completed industrial implementation, but propose a formal scaffold and explain why it defines the right class of problems for further implementation work.
\end{abstract}

\noindent\textbf{Keywords:} controlled generative development; structural realization; output contract; pre-code verification; verification-driven dialogue; task-driven context assembly; abstraction control; AI-native software design

\section{Introduction}

Generative models have dramatically reduced the cost of obtaining code, but they have not solved an earlier and more fundamental problem: how to preserve the causal link between \textbf{what the author wanted}, \textbf{how it was formalized}, and \textbf{what is ultimately executed}. In a typical workflow, a user writes a free-form request, the model returns a mixture of text, pseudocode, lists, and diagrams, and the developer manually tries to reconstruct the architecture of the solution. As a result, the cheapest defects to fix---missing branches, absent input artifacts, implicit contracts, and broken data flows---are discovered too late, after they have already been embodied in code.

Modern agentic systems such as OpenAI Symphony (2026) \cite{openai2026symphony} automate the transition from task to code, but in the public descriptions currently available for such systems, no intermediate verifiable layer is singled out between intent and execution. This work addresses precisely that gap.

The problem is not simply that models are ``not intelligent enough.'' The deeper problem is that between intent and code there is often no stable layer suitable for formal work. A free-form model response may be useful to a person as a sketch, but it is inconvenient as an object of analysis: its form drifts, its completeness varies, its level of abstraction is unstable, and many assumptions remain implicit. This is why such a setup makes it difficult to compare solutions reliably, assemble context for a specific change, detect defects before generation, and hand the same specification to different execution backends.

This paper shifts the center of gravity. We do not place at the center the question ``which model is better,'' and we do not search for a single correct graph. Instead, we fix the layer into which a model's response must be brought \textbf{before} code generation. We call this layer \textbf{structural realization}. It is neither a free-form description of intent nor executable code, but it is precisely the place where one can: verify the task before code generation; make omissions and contradictions observable; assemble relevant context for the next change; compare and correct solutions without reducing the paper to a competition between models; and hand the same specification to different execution backends.

In this sense, the key shift of the work lies not in making the input more elaborate, but in \textbf{fixing the output}. We argue that practical progress in AI-assisted development begins not where the user writes increasingly clever prompts, but where the model response is required to fit into a verifiable and controllable structure in which incompleteness becomes observable.

To this end we propose the \textbf{Controlled Generative Development (CGD)} framing. At the top level it is described by the chain \textbf{Intent $\to$ Structural Realization $\to$ Execution}. Architecturally, before handoff to a backend, a project is described by the pair $\mathbf{D = \langle I, R \rangle}$, where $I$ is an intent representation at the interface level and $R$ is the structural realization. For CGD, the formal core of the second element is given by the triple $\mathbf{R = \langle G, \Sigma, C \rangle}$, where $G$ is a typed graph of functions and tables, $\Sigma$ is a set of table signatures, and $C$ is a set of function contracts.

The main contribution of the paper is neither the claim that we have found the ``one correct graph,'' nor a comparison of specific models against each other. We show something else: if intent is translated into a strict and verifiable structural layer in which omissions become observable, then the mixing of problems disappears. Questions of response form, level of abstraction, completeness of description, data compatibility, and the choice of execution backend no longer live at the same level; they begin to be solved where they properly belong.

\section{Three control loops: output contract, abstraction control, verification profile}

\subsection{Contract-first at the communication level}

CGD implements contract-first not only at the level of functions and tables, but also at the level of interaction with the model. The model receives not merely a task, but a \textbf{contract on the shape of the response}. In this formulation, the central object becomes not the prompt content itself, but the space of admissible responses into which the model must fit.

This shift matters in practice. As long as the response format is not fixed, the user and the tool work with text meant for reading. Once the response format becomes part of the contract, the response turns into data meant for analysis. A missing table, an unfilled contract, an incomplete error path, or a drifting abstraction level become observable defects rather than something to be guessed at.

\subsection{Output contract}

The first control loop is the \textbf{output contract}. It defines \textbf{what} must be present in the model response and \textbf{how} it is serialized.

Inside it, it is useful to distinguish two substructures. The first answers the question \textbf{what is required}: which components must exist in the response at all. In the CGD context, these are functions, tables, signatures, contracts, and typed relationships. The second answers the question \textbf{how it is represented}: which machine-readable and human-readable forms are used to carry the result. For example, one part may be serialized as a structured JSON profile, while another part may be expressed as a human-readable graph projection.

This distinction is fundamental. It makes it possible to avoid conflating the substantive obligations of the response with the concrete format used to represent them. The output contract is not equal to a single ``file format''; it combines the mandatory components of the response and the way they are serialized.

The concrete realization of the output contract for CGD---the CGD Response Standard---is given in Appendix~\ref{app:response-standard}.

\subsection{Abstraction control}

The second control loop is \textbf{abstraction control}. Even when the form of the response is fixed, the model can still vary dramatically in its level of detail.

One response may split a solution into many micro-steps, another may compress everything into a single large node, and a third may mix architectural and operational levels. For this reason, fixing structure and form alone is not enough. One must also constrain the \textbf{resolution} of the response: top-level architecture, detailed decomposition, local workflow, internal function step, and so on.

An important effect of abstraction control is that it does not force the model into a single architecture. It constrains only the admissible \textbf{level of description}, while leaving the value axis free.

The concrete realization of abstraction control for CGD---the Execution Profile---is given in Appendix~\ref{app:execution-profile}.

\subsection{Verification profile}

The third control loop is the \textbf{verification profile}. It determines which properties of the response are checked before code generation, which defects count as blocking, what may be repaired automatically, and at what point the structural realization may be considered sufficiently complete for transfer to a backend.

The verification profile completes the transition from ``text to read'' to ``structure to work with.'' The output contract makes the response inspectable, abstraction control makes it comparable in depth, and the verification profile makes it suitable for formal pre-code verification.

\subsection{Why these three loops are sufficient for the first version}

These three loops do not attempt to prove that an architecture is optimal, nor do they guarantee the semantic correctness of the solution. Their task is more modest, but more fundamental: to eliminate the chaos of mixed levels and translate the model response into a space in which completeness can be observed, consistency can be checked, corrective questions can be asked, and the resulting structure can be reused in further development.

That is why CGD starts not from the search for an ideal model, but from the design of a controllable response space.

\subsection{Positioning}

By early 2026, the spec-first approach had become a stable industrial trend: academic surveys systematize levels of specification strictness \cite{piskala2026sdd}, while industrial tools---from CodeSpeak \cite{breslav2026codespeak} and Augment Intent \cite{augment2026intent} to OpenAI Symphony \cite{openai2026symphony}, Cursor Rules \cite{cursor2026rules}, Claude Code \cite{anthropic2026claudecode}, and GitHub Spec Kit \cite{github2025speckit}---treat specification as the primary development artifact rather than as a free-form prompt.

However, all of these systems address primarily one side of the problem: the transition from specification to code. In the publicly available descriptions of these systems, no separate intermediate verifiable layer is singled out in which the specification itself can be checked, structural defects can be detected, and the system can be brought to specification closure \textbf{before} code generation. CGD addresses precisely this gap: the formal core $R = \langle G, \Sigma, C \rangle$, the output contract, abstraction control, and the verification profile define a space in which an error is caught before it becomes code.

\section{Architectural framing: $D = \langle I, R \rangle$}

\subsection{Intent and Structural Realization}

The top-level CGD scheme is not a flat set of equal components, but an architectural cause $\to$ consequence pair:


$$D = \langle I, R \rangle.$$

The first element, $I$, represents \textbf{intent}. This is where the user or author describes what needs to be done. In our current implementation, the interface level is served by \textbf{YASF (Yet Another Structured Format)}---a minimal author-facing format for describing intent. At this level, the task is fixed in terms of the problem domain and in names that are natural for the author; these names and structures do not yet need to coincide with the future structural realization. YASF is not a formal realization and does not directly define $G$, $\Sigma$, or $C$; it merely translates a free-form description into a structured interface representation suitable for further compilation. The transition from the first element of the pair to the second consists in compiling a YASF description into the structural realization $R = \langle G, \Sigma, C \rangle$, where the graph, table signatures, and function contracts already obey strict rules of consistency and verification. The full YASF specification is given in Appendix~\ref{app:yasf}.

The second element, $R$, represents the \textbf{structural realization} of intent. This is no longer a free-form description, but a formal model that can be processed automatically: verified, analyzed, modified, and handed to an execution system.

\subsection{Why the pair is needed but does not replace the formal core}

The pair $D = \langle I, R \rangle$ answers the question \textbf{where the boundary lies between intent and formalized realization}. It is not meant to express all details of verification. Formal work begins inside the second element.

For CGD, this second element has a concrete internal structure:


$$R = \langle G, \Sigma, C \rangle.$$

The pair therefore does not replace the triple. It puts it in the right place. Without this distinction, it is easy to mix the language of intent, the standards that represent the response, and the realization model itself. In the new formulation, these confusions disappear: $D$ explains the architectural framing, while $R$ defines the formal core of CGD.

\subsection{Intent $\to$ Structural Realization $\to$ Execution}

The CGD architectural chain has three links. At the \textbf{Intent} level, the task is fixed in an interface form convenient for the author. At the \textbf{Structural Realization} level, the intent is translated into formal structures suitable for verification and analysis. At the \textbf{Execution} level, the structural realization is handed to an execution system: a code generator, interpreter, simulator, analytical backend, or multi-agent system.

This chain makes it possible to separate three tasks that are often mixed together: describing meaning, formalizing that meaning, and executing it. That separation is the main architectural gain of the new version of the paper.

\section{The formal core of CGD: $R = \langle G, \Sigma, C \rangle$}

\subsection{Graph $G$}

The graph $G$ is defined as a typed directed graph


$$G = (V, E, \mathrm{src}, \mathrm{dst}, \tau_V, \tau_E),$$

where the node type satisfies $\tau_V(v) \in \{\texttt{function}, \texttt{table}\}$, and the edge type satisfies $\tau_E(e) \in \{\texttt{consumes}, \texttt{produces}, \texttt{reads}, \texttt{writes}, \texttt{triggers}\}$.

The basic admissible combinations are: \texttt{consumes}: table $\to$ function; \texttt{produces}: function $\to$ table; \texttt{reads}: table $\to$ function; \texttt{writes}: function $\to$ table; \texttt{triggers}: function $\to$ function.

The graph fixes the architecturally meaningful topology of processing: which functions participate in the process, which tables act as inputs, outputs, and states, and which dependencies between steps are trigger dependencies rather than data flows.

\subsection{Table signatures $\Sigma$}

A table signature $\Sigma(t)$ is defined as


$$\Sigma(t) = \langle \mathrm{Fields}, \mathrm{Keys}, \mathrm{Kind}, \mathrm{Constraints} \rangle.$$

Here \texttt{Fields} describes fields and their types, \texttt{Keys} describes primary, unique, and foreign keys, \texttt{Kind} describes the role of the table in the model (\texttt{entity}, \texttt{event}, \texttt{reference}, \texttt{log}, \texttt{projection}, \texttt{error}), and \texttt{Constraints} captures additional integrity constraints.

Signatures are not a duplicate of a relational schema for its own sake. They are the source of type information for checking compatibility between graph relations and function contracts. If the graph says that a function produces a table, while the signatures show that the table is incompatible with the expected data structure, that must be detected before code generation.

\subsection{Function contracts $C$}

A function contract $C(f)$ is defined as


$$C(f) = \langle C_{\mathrm{in}}, C_{\mathrm{out}}, R, W, T_{\mathrm{in}}, T_{\mathrm{out}} \rangle,$$

where: $C_{\mathrm{in}}$~--- the tables consumed by the function; $C_{\mathrm{out}}$~--- the tables created by the function; $R$~--- the tables read by the function; $W$~--- the tables written by the function; $T_{\mathrm{in}}$~--- incoming trigger dependencies; $T_{\mathrm{out}}$~--- outgoing trigger dependencies.

Inside the contract, the symbol $R$ denotes the reads component of a function; in the pair $D = \langle I, R \rangle$, the same symbol denotes the structural realization as a whole. The distinction is given by context.

A function contract links the graph and the signatures: it describes not merely the fact that a \texttt{function} node exists, but how that node interacts with the data structure and with other processing steps.

\subsection{Root property: verification before code generation}

The main property of the triple $\langle G, \Sigma, C \rangle$ is that it makes it possible to catch errors \textbf{before} code generation. At this level one can already detect: missing input artifacts; omitted processing branches; broken data flows; incomplete contracts; incompatible table signatures; isolated or unreachable elements; and mixed abstraction levels within one response.

This is the root property of CGD. Structural realization is needed neither for a pretty diagram nor to duplicate code. It is needed because this is the layer where an error is still cheap to fix.

\subsection{The triune task: execute, analyze, modify}

It is precisely the structural realization---and not intent and not code---that is capable of solving three tasks at once.

\textbf{Execute.} The triple $\langle G, \Sigma, C \rangle$ can be handed to an execution system. \textbf{Analyze.} It supports formal checking of consistency, completeness, and compatibility. \textbf{Modify.} It supports controlled transformations: changing nodes and links, updating contracts, restructuring data, and shifting the level of detail.

Intent is too free-form to solve all three tasks reliably. Code is too detailed and too tightly bound to a chosen backend. Only the intermediate structural layer is both sufficiently strict and sufficiently compact.

\subsection{Specification as code with legibility}

Structural realization has one more important property: it behaves like code while remaining more legible than code. It can be versioned, diffed, validated, passed between tools, and used as a source for generation. At the same time, it remains readable at the architectural level: the graph shows topology, signatures show data, and contracts show processing interfaces.

This effect is conveniently described as \textbf{specification as code with legibility}.

\subsection{Contract-preserving transformations}

The triple $\langle G, \Sigma, C \rangle$ supports contract-preserving transformations: local decomposition of a subgraph with preservation of its external contract, and reverse aggregation. For example, three micro-functions with compatible contracts can be collapsed into one macro-function if the external contract of the subgraph matches the contract of that macro-function. For the first version of the paper, what matters is to fix the very existence of such transformations, because this explains why CGD is able not only to store structure, but also to work with different levels of its detail. The full algorithm for computing a subgraph contract and the formal rules for such transformations are moved to Appendix~\ref{app:subgraph-contract}.

\section{Verification-driven dialogue and specification closure}

\subsection{Verification profile as an operating mechanism}

In CGD, the verification profile is not an abstract idea of ``checking something before code.'' It is a concrete operating mechanism that combines automatic checks, defect classification, and targeted dialogue with the author.

As soon as a model response has been brought to the triple $\langle G, \Sigma, C \rangle$, the system is able not only to validate format, but also to ask substantive questions about incompleteness in the structure.

\subsection{Classes of defects}

Defects are divided into three classes.

\textbf{Blocking.} Defects that cannot be eliminated without a decision from the author. For example, a missing error path, an undefined input artifact, or a fundamentally incomplete function contract.

\textbf{Auto-resolvable.} Defects that can be corrected automatically without changing semantics. For example, name normalization, alias merging according to a predefined rule, canonicalization of type notations, or deletion of structural duplicates.

\textbf{Advisory.} Remarks that do not block progress, but point to a potential problem: unconfirmed redundancy, an optional downstream step left uncovered, an unused success event, incomplete logging, and so on.

This separation matters because it does not mix technical normalization, substantive omissions, and architectural recommendations.

\subsection{Verification-driven dialogue}

When the verifier detects a blocking defect, it does not attempt to silently ``guess on behalf of the author.'' Instead, it generates a \textbf{targeted question}, formulated in natural language. The author's response updates the structural realization, after which verification is launched again.

This loop is what we call \textbf{verification-driven dialogue}. It moves verification from the mode ``the model produced text and the human noticed the problem by hand'' to the mode ``the system localizes the defect, asks a question where automatic repair is not admissible, and resumes the process after receiving an answer.''

\subsection{Specification closure}

A specification is considered to have reached \textbf{specification closure} when all of the following conditions are satisfied:

\begin{enumerate}
\item the structural realization is consistent: graph, signatures, and contracts do not contradict one another;
\item all mandatory contract components are defined;
\item no unresolved blocking defects remain;
\item the mandatory processing paths, including error handling, have explicit structural expression;
\item all advisory remarks have either been eliminated or explicitly accepted as admissible within the current verification profile.
\end{enumerate}

Importantly, specification closure is neither a property of a single level nor a sign that the architecture is ``ideal.'' It is a \textbf{sufficient state} for moving to the next stage: code generation, simulation, or another execution-oriented processing step.

\section{Task-driven context assembly}

In CGD, task-driven context assembly is not performed over the entire chat history and not through general embedding similarity. It is driven by the affected objects of the structural realization. When a new task arrives, the system first localizes it in $R = \langle G, \Sigma, C \rangle$: it finds the functions, tables, contracts, and nearest dependency neighborhood touched by the change. Those very objects are then used as keys for searching for additional information in the history of interaction---deferred decisions, author preferences, disputed areas, and rules that have not yet been fully materialized in the current realization.

For example, if the task is ``add error logging for registration,'' the system locates the registration function, its contract, the tables on the error path, and the nearest logging sink. It then lifts the history for those same objects: it may turn out that two weeks earlier the author already recorded a preference to log errors separately from the success path, but that decision was never introduced into the structure. The resulting context includes the current state of the affected objects, their local neighborhood, and the historical decision that was found. This makes the context compact, current, and anchored to the single source of truth---the structural realization; the full formalization is moved to Appendix~\ref{app:context-assembly}.

\section{Demo 1---controlling form}

\subsection{Setup}

The same user registration request is taken twice. In the first run it is submitted without an output contract. In the second run it is submitted with an explicit output contract that requires the response to return functions, tables, signatures, contracts, and a typed projection of relations in a fixed form.

To demonstrate the effect of form, responses from different models are used; comparing models against one another is not the goal of this demo. By contrast, Demo~2 uses the same model in order to isolate the effect of one factor only---the level of abstraction.

\subsection{Real baseline fragment}

The following is a verbatim fragment from a Claude Opus 4.6 baseline response without a structuring output contract:

\begin{lstlisting}
## Functions

### 1. register_user(email, password) -> User

Creates a user and starts confirmation.

Input:  email: str, password: str
Output: User (id, email, is_verified, created_at)
Errors: ValueError -- email already taken or invalid

Logic:
  1. validate_email(email)
  2. hash = hash_password(password)
  3. INSERT INTO users (email, password_hash)
  4. token = create_verification_token(user.id)
  5. send_verification_email(email, token)
  6. log_action(user.id, 'register', {ip})
  7. return user
\end{lstlisting}

This fragment is illustrative because it mixes the function interface, logic, errors, and side effects, but contains no explicit separation between graph, data, contract, and downstream dependencies. Omissions remain implicit: one cannot formally check whether a separate input artifact exists, how the error path is materialized, or which tables act as outputs of the processing.

The point of the first demonstration is not that the output contract makes the model ``smarter.'' It makes the response \textbf{workable}. The response ceases to be text for reading and becomes structure for analysis.

\section{Demo 3---verification-driven correction}

\subsection{Setup}

In the third demo, one of the structured responses obtained after Demo~1 and Demo~2 is passed through the verification profile. The goal of the demonstration is not to drive the response to a rigid predefined canon, but to show how structured output becomes an object of verification and correction until specification closure is reached.

The full artifacts of the demo runs are given in Appendix~\ref{app:demo-artifacts}.

\subsection{First verifier pass}

In the real GPT-5.4 run, the main handler \texttt{RegisterUser} already has an explicit input artifact $C_{\mathrm{in}} = [\text{``RegistrationRequest''}]$, so the defect does not lie in the main registration step. The verifier detects two blocking defects:

\begin{enumerate}
\item \textbf{Blocking:} the function \texttt{SendConfirmationEmail} has an empty $C_{\mathrm{in}}$, even though the email-sending step depends on an input artifact that is not structurally expressed.
\item \textbf{Blocking:} the registration flow lacks a materialized error path: the error is mentioned at the semantic level, but receives no separate structural expression in the graph, signatures, and contracts.
\end{enumerate}

Besides the blocking defects, the same run contains auto-resolvable notational deviations: a non-standard form of Mermaid edge labels and values such as \texttt{format: "text"} / \texttt{format: "bool"} in JSON signatures. These deviations can be normalized automatically and do not require a decision from the author.

\subsection{Dialogue with the author}

For both blocking defects, the system asks targeted questions: which object should count as the explicit input artifact for \texttt{SendConfirmationEmail}; and how the registration error should be materialized in the structural realization: as a separate table, an event, or another explicitly specified sink.

The author's answer updates the triple $\langle G, \Sigma, C \rangle$: an explicit $C_{\mathrm{in}}$ is added to the contract of \texttt{SendConfirmationEmail}, and the error path receives structural expression with recomputation of the related signatures and contracts.

\subsection{Re-check and closure}

The expected result of the second verification pass is as follows: after adding an explicit input artifact to the contract of \texttt{SendConfirmationEmail} and materializing the error path, both blocking defects can be eliminated. In the current version of the paper, this second pass is not shown as a separate saved artifact; here we fix the structure of the correction and the condition under which specification closure is reached.

\subsection{Takeaway from Demo 3}

The third demonstration closes the main logic of the paper. The output contract provides structure, abstraction control keeps the description at the intended level, the verification profile localizes defects, and verification-driven dialogue provides a mechanism by which a structural realization can be brought to specification closure. In the current demo, the first verification pass and the structure of the correction are shown; an empirical record of the full cycle up to closure is the nearest next step.

\section{Limitations and threats to validity}

\subsection{Limits of the claims}

This work is conceptual and architectural. We do not claim that we have already built a complete reference implementation, nor do we claim demonstrated industrial effectiveness. The paper shows how the problem should be structured rather than presenting a final operational report.

Moreover, the output contract does not guarantee an optimal architecture. It guarantees something else: it makes an architecture \textbf{observable, verifiable, and correctable before code generation}.

\subsection{Limits of the model}

The formal core of CGD primarily covers the class of systems with discrete data flows and explicit dependencies between functions and tables. The full operational semantics of branching, loops, concurrency, timing, and transactionality are not exhausted in this version of the paper.

\subsection{Threats to construct validity}

The danger here is that the chosen control loops may look sufficient for a broader class of tasks than they actually cover. We reduce this risk by explicitly distinguishing the architectural framing $D = \langle I, R \rangle$ from the formal core $R = \langle G, \Sigma, C \rangle$, and by not extending the claims to every possible class of computational systems.

\subsection{Threats to internal validity}

Internal validity is limited by the absence of a full reference implementation. The demonstrations shown support the architectural plausibility of the approach, but do not prove that every concrete implementation of a verification profile or task-driven context assembly will necessarily yield the same practical effect.

\subsection{Threats to external validity}

External validity is limited by the fact that the paper uses tasks that naturally fit a processing graph and data tables. Extending the approach to other classes of systems will require additional representation profiles and possibly an extension of the triple $\langle G, \Sigma, C \rangle$.

\subsection{Threats to reliability}

The reliability of results depends on the stability of the output contract, the normalization rules, and the verification profile. If these rules remain implicit or drift between runs, the paper loses one of its main advantages---repeatability at the structural level. For this reason, the output contract, the verification profile, and the normalization rules must be versioned on a par with the structural realization itself. This is precisely why we move strictness from the domain of ``good prompts'' into the domain of explicit profiles and verification rules.

\section{Future Work}

\subsection{Calibrating the output contract and abstraction control}

The first natural direction for future work is to calibrate the output contract and abstraction control against fixed structural metrics. Once the form of the response has been standardized, prompt and profile optimization becomes possible against criteria such as completeness, consistency, and suitability for downstream verification.

\subsection{Classes of equivalent solutions}

The second important task is to define classes of equivalent solutions. For practical work, it is not enough to distinguish ``literally identical'' from ``not identical.'' Formal criteria are needed to distinguish a different level of detail, an admissible structural transformation, and a genuine architectural divergence.

\subsection{Graph transformations and path-based comparison}

The third direction is graph transformations and path-based comparison. Several related questions open here at once: admissible collapses and expansions of subgraphs, contract-preserving transformations, comparison of realizations by the set of paths from inputs to outputs, and determination of which differences belong to form and which belong to architectural substance.

\subsection{Extending the set of execution backends}

The fourth direction concerns the backend level. One and the same structural realization should be suitable not only for code generation, but also for interpretation, simulation, analytics, and agentic execution. Determining which properties of the triple $\langle G, \Sigma, C \rangle$ must remain invariant across such backends is an interesting research problem in its own right.

\section{Conclusion}

This paper has proposed a new formulation of the problem for AI-native development. Instead of treating prompt and code as two endpoints of the process, we introduce an intermediate layer---\textbf{structural realization}---that becomes the main object of verification, analysis, and transfer to a backend.

At the top level, CGD is defined by the architectural chain \textbf{Intent $\to$ Structural Realization $\to$ Execution} and by the pair $\mathbf{D = \langle I, R \rangle}$. Inside CGD, structural realization is defined by the triple $\mathbf{R = \langle G, \Sigma, C \rangle}$. This separation removes the mixing of levels: the interface-level description of intent, the formal realization layer, and the execution system no longer compete for the same place in the model.

The three control loops---\textbf{output contract}, \textbf{abstraction control}, and \textbf{verification profile}---translate a model response from the space of free text into the space of controlled structure. The output contract makes incompleteness observable, abstraction control keeps the admissible depth of description under control, and the verification profile makes it possible to ask targeted questions and bring the realization to specification closure.

The formal core $\langle G, \Sigma, C \rangle$ provides the root property of CGD: verification before code generation. It is precisely on this layer that the triune task becomes possible---to execute, analyze, and modify one and the same specification. Task-driven context assembly defines the principle according to which this layer is needed not only for verification, but also for assembling relevant context for the next change. Finally, the architectural framing is designed so as to admit backend substitutability: one and the same structural realization is, in principle, suitable for transfer into code generation, interpretation, simulation, or a multi-agent system. Verifying this property in practice is a separate direction of future work (\S11.4).

We do not claim that this already solves the problem of correct program-system generation. But we do claim that this way of structuring the problem formulates it correctly. It distributes intent, formalization, and execution across their proper levels and thereby removes the mutual interference of problems that, in ordinary AI-assisted development, end up mixed in one place.
