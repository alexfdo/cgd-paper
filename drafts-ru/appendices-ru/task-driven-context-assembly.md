# Appendix D — Task-Driven Context Assembly

## D.1. Назначение

Этот appendix фиксирует черновую формализацию `task-driven context assembly` как механизма сборки контекста не по всей истории диалога, а по затронутым объектам структурной реализации.

## D.2. Цель

Цель механизма — получить компактный и релевантный контекст для следующей задачи, используя `R = ⟨G, Σ, C⟩` как единственный источник структурной локализации.

## D.3. Вход

На вход подаётся:
- новая задача `q`;
- текущая структурная реализация `R = ⟨G, Σ, C⟩`;
- история решений `H`, содержащая предыдущие уточнения, принятые допущения и нерешённые вопросы.

## D.4. Локализация задачи в `R`

Сначала задача `q` локализуется в структурной реализации. Результатом этого шага является стартовое множество затронутых объектов:

`A₀(q) = F₀ ∪ T₀`,

где:
- `F₀` — функции, явно упомянутые или семантически соответствующие задаче;
- `T₀` — таблицы, явно упомянутые или напрямую связанные с найденными функциями.

Практически это означает привязку задачи к уже существующим узлам графа, их контрактам и сигнатурам, а не к общей истории чата.

## D.5. Выбор локальной окрестности

После локализации строится локальная структурная окрестность `N(A₀)`.

В минимальной версии в неё входят:
- сами затронутые функции и таблицы;
- таблицы из `Cin`, `Cout`, `R`, `W` для найденных функций;
- функции из `Tin`, `Tout`;
- соседние узлы на расстоянии одного шага по графу зависимостей.

Идея проста: в контекст попадает не весь граф, а только ближайшая структурная зона, в которой изменение реально может повлиять на поведение системы.

## D.6. Подъём истории по структурным ключам

После построения `N(A₀)` история `H` поднимается не целиком, а только по структурным ключам из найденной окрестности.

Отбираются:
- прошлые решения, явно относящиеся к функциям и таблицам из `N(A₀)`;
- ранее зафиксированные blocking- или advisory-замечания по тем же объектам;
- принятые допущения, которые ещё не были материализованы в текущей версии `R`.

Таким образом, история становится не самостоятельным источником истины, а вспомогательным слоем, индексируемым по объектам структурной реализации.

## D.7. Сборка task-specific context

Итоговый task-specific context `CTX(q)` состоит из трёх частей:
- текущего состояния затронутых объектов;
- локальной структурной окрестности;
- релевантных исторических решений, поднятых по тем же ключам.

То есть:

`CTX(q) = Current(A₀ ∪ N(A₀)) ∪ History(A₀ ∪ N(A₀))`.

В контекст не включаются несвязанные фрагменты графа и несвязанная история, даже если они тематически похожи на задачу.

## D.8. Выход

На выходе получается компактный структурно привязанный контекст, пригодный для:
- модификации `R`;
- повторной верификации;
- подготовки следующего модельного шага.

## D.9. Иллюстративный пример

Если приходит задача «добавить журналирование ошибок регистрации», механизм локализует её в функции регистрации, её error-path и ближайшем logging sink. Затем по тем же объектам поднимается история: может оказаться, что ранее уже было принято решение логировать ошибки отдельно от success-path, но это решение не было внесено в текущую реализацию. В task-specific context попадает именно эта локальная зона, а не вся переписка и не вся система.

## D.10. Статус

В текущей статье `task-driven context assembly` вводится как архитектурный принцип и черновая формализация. Полный алгоритм выбора окрестности, ранжирования исторических фрагментов и оценки достаточности контекста остаётся предметом дальнейшей работы.
