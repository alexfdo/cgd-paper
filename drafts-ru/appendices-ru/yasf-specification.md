# Appendix F — Спецификация YASF (Yet Another Structured Format)

## F.1. Назначение

YASF — детерминированный язык спецификации, используемый в текущей реализации CGD как форма интерфейсного представления намерения **I** в архитектурной двойке **D = ⟨I, R⟩**. YASF описывает систему на контролируемом естественном языке с каноническим набором секций. Текст YASF компилируется в структурную реализацию **R = ⟨G, Σ, C⟩**, где **G** — типизированный граф функций и таблиц, **Σ** — сигнатуры таблиц, **C** — контракты функций. YASF выполняет роль авторского слоя: это формат, в котором человек фиксирует намерение до перевода в формальное ядро.

## F.2. Проектные принципы

YASF спроектирован на основе следующих принципов:

Принцип 1 — контролируемый естественный язык. Текст YASF пишется на естественном языке, но с фиксированной структурой секций и терминологией. Это обеспечивает низкий порог входа при сохранении компилируемости.

Принцип 2 — детерминированность компиляции. Для одного и того же согласованного состояния текста YASF компилятор порождает одну и ту же структурную реализацию **R = ⟨G, Σ, C⟩** с теми же графовыми связями, контрактами и сигнатурами. Детерминированность является проектным инвариантом и верифицируется через conformance tests (фиксированные пары «вход YASF → эталонная структурная реализация»).

Принцип 3 — минимальность. YASF описывает ровно то, что необходимо для построения структурной реализации: функции, таблицы, потоки данных, ограничения и контракты. Детали реализации, такие как алгоритмы, структуры данных и стиль кода, не входят в YASF.

Принцип 4 — модульность. Каждая функция описывается отдельным блоком YASF. Блоки могут ссылаться друг на друга через имена функций и таблиц. Это обеспечивает независимую разработку, верификацию и компиляцию отдельных частей намерения.

Принцип 5 — прослеживаемость. Каждый элемент структурной реализации — узел или ребро графа **G**, сигнатура **Σ(t)** и контракт **C(f)** — должен быть прослежен до конкретной секции текста YASF. Обратно: каждая секция YASF должна порождать определённые элементы в **R = ⟨G, Σ, C⟩**.

## F.3. Структура блока YASF

Блок YASF описывает одну функцию. Каноническая структура блока включает следующие секции:

Назначение — краткое описание функции на естественном языке (1–2 предложения). Компилируется в: узел `function` в графе **G** с именем и описанием.

Входные данные — перечисление таблиц, потребляемых функцией, с указанием полей, типов, ключей и `kind`. Компилируется в: узлы `table` в **G**, рёбра `consumes`, компонент `Cin` контракта и сигнатуры **Σ(t)** входных таблиц.

Справочные данные — перечисление таблиц, читаемых функцией для справки. Компилируется в: рёбра `reads`, компонент `R` контракта и, при необходимости, сигнатуры соответствующих таблиц.

Обработка — описание последовательности шагов обработки на контролируемом естественном языке. Порядок элементов задаёт базовую последовательность шагов. Конкретные рёбра графа и зависимости в контрактах формируются при компиляции совместно с сигнатурами таблиц и другими секциями блока.

Выходные данные — перечисление таблиц, производимых функцией, с указанием полей, типов, ключей и `kind`. Компилируется в: рёбра `produces`, компонент `Cout` контракта и сигнатуры **Σ(t)** выходных таблиц.

Ошибки — перечисление таблиц ошибок, производимых функцией при отклонении от `success-path`. Компилируется в: рёбра `produces`, дополнительные элементы `Cout` и сигнатуры **Σ(t)** таблиц ошибок (`kind: error`).

Побочные эффекты — перечисление таблиц, в которые функция записывает данные, не являющиеся её основным выходом. Компилируется в: рёбра `writes`, компонент `W` контракта и при необходимости сигнатуры соответствующих таблиц.

Триггеры — перечисление функций, активируемых данной функцией по завершении. Компилируется в: рёбра `triggers`, компонент `Tout` контракта.

Контракт — явная фиксация шестикомпонентного контракта **C(f) = ⟨Cin, Cout, R, W, Tin, Tout⟩**. Может быть записан автором вручную или сгенерирован компилятором из остальных секций. При наличии обоих компилятор проверяет согласованность.

## F.4. Правила компиляции YASF → R = ⟨G, Σ, C⟩

Правило 1 — извлечение узлов. Каждое имя функции, упомянутое в секциях «Назначение» или «Триггеры», порождает узел с `τ_V = function`. Каждое имя таблицы, упомянутое в секциях «Входные данные», «Справочные данные», «Выходные данные», «Ошибки», «Побочные эффекты», порождает узел с `τ_V = table`.

Правило 2 — извлечение рёбер. Таблица из секции «Входные данные» порождает ребро `consumes (table → function)`. Таблица из секции «Выходные данные» или «Ошибки» порождает ребро `produces (function → table)`. Таблица из секции «Справочные данные» порождает ребро `reads (table → function)`. Таблица из секции «Побочные эффекты» порождает ребро `writes (function → table)`. Функция из секции «Триггеры» порождает ребро `triggers (function → function)`.

Правило 3 — извлечение сигнатур. Для каждой таблицы, описанной с полями, типами и ключами, формируется сигнатура **Σ(t) = ⟨Fields, Keys, Kind, Constraints⟩** по определению формального ядра статьи.

Правило 4 — формирование контракта. Из извлечённых рёбер формируется контракт **C(f)**. Если автор указал контракт в секции «Контракт» явно, компилятор проверяет совпадение сгенерированного и авторского контрактов. Расхождение считается структурным дефектом.

Правило 5 — проверка полноты. После компиляции блока проверяется: все таблицы в контракте имеют определённые сигнатуры, все рёбра типово корректны, нет узлов без рёбер, а полученная локальная структурная реализация согласуется с verification profile текущего шага.

## F.5. Связь секций YASF с компонентами контракта

| Секция YASF | Компонент контракта | Тип ребра | Направление |
| --- | --- | --- | --- |
| Входные данные | Cin | consumes | table → function |
| Справочные данные | R | reads | table → function |
| Выходные данные | Cout | produces | function → table |
| Ошибки | Cout | produces | function → table |
| Побочные эффекты | W | writes | function → table |
| Триггеры | Tout | triggers | function → function |
| Внешний блок, ссылающийся на данную функцию в своих триггерах | Tin | triggers | function → function |

Компонент `Tin` не имеет собственной секции в блоке функции: он выводится из секций «Триггеры» других блоков, ссылающихся на данную функцию.

## F.6. Пример блока YASF

Функция: RegisterUser  
Назначение: Регистрация нового пользователя по email.

Входные данные:  
  RegistrationRequest (id PK, name: string required, email: string required,  
                       timestamp: timestamp required; kind: event)

Справочные данные:  
  User (id PK, name: string, email: string, status: string,  
        created_at: timestamp; kind: entity)  
  — читается для проверки уникальности email.

Обработка:  
  1. Принять RegistrationRequest.  
  2. Проверить формат и полноту данных.  
  3. Проверить уникальность email по таблице User.  
  4. Создать запись User со статусом pending.  
  5. Зафиксировать RegistrationEvent.  
  6. Записать действие в AuditLog.  
  7. Инициировать SendConfirmation.

Выходные данные:  
  User (id PK, name: string, email: string, status: string,  
        created_at: timestamp; kind: entity)  
  RegistrationEvent (id PK, user_id FK→User, event_type: string,  
                     timestamp: timestamp; kind: event)

Ошибки:  
  RegistrationError (id PK, error_code: string, message: string,  
                     timestamp: timestamp; kind: error)  
  — при невалидном вводе или дублировании email.

Побочные эффекты:  
  AuditLog (id PK, action: string, entity_id: string,  
            timestamp: timestamp; kind: log)

Триггеры:  
  SendConfirmation  
  — отправка подтверждения регистрации.

Контракт v2:  
  C(RegisterUser) = ⟨Cin: {RegistrationRequest},  
                      Cout: {User, RegistrationEvent, RegistrationError},  
                      R: {User},  
                      W: {AuditLog},  
                      Tin: ∅,  
                      Tout: {SendConfirmation}⟩

## F.7. Декомпозиция в YASF

При декомпозиции функции на подфункции каждая подфункция получает собственный блок YASF. Исходный блок сохраняется как агрегированное представление с пометкой:

Функция: RegisterUser (агрегированный узел)  
Декомпозиция: ValidateInput → CheckUniqueness → CreateUser → (triggers) SendConfirmation  
Refinement map: см. блоки ValidateInput, CheckUniqueness, CreateUser, SendConfirmation

Каждый блок подфункции имеет полную структуру §F.3. Промежуточные таблицы (`ValidatedRequest`, `UniqueRequest`) описываются в секциях «Выходные данные» и «Входные данные» соответствующих подфункций. Формальное вычисление внешнего контракта подграфа при такой декомпозиции вынесено в Appendix C.

## F.8. Verification-driven dialogue в YASF

Когда верификатор обнаруживает дефекты после компиляции блока YASF, они фиксируются и классифицируются:

Blocking — требуют дополнения текста YASF автором. Пример: секция «Ошибки» отсутствует или пуста при наличии условий ветвления в секции «Обработка». Верификатор формирует вопрос; после ответа автора секция дополняется.

Auto-resolvable — компилятор или верификатор может дополнить блок автоматически. Пример: если `error-path` добавлен и функция уже имеет побочный эффект `AuditLog`, запись в `AuditLog` на `error-path` добавляется по аналогии.

Advisory — предупреждения, фиксируемые как комментарии к блоку YASF. Пример: таблица в «Выходных данных» не потребляется ни одной известной функцией.

После устранения всех blocking-дефектов и применения auto-resolvable исправлений блок перекомпилируется в **R = ⟨G, Σ, C⟩** и проверяется повторно до достижения specification closure (см. §5 основного текста).

## F.9. Conformance tests

Детерминированность компиляции верифицируется через conformance tests — фиксированный набор пар:

| Вход | Ожидаемый выход |
| --- | --- |
| Текст YASF (блок функции) | Подграф **G**: узлы, рёбра, типы |
| Текст YASF (блок функции) | Контракт **C(f)**: шесть компонент |
| Текст YASF (описание таблицы) | Сигнатура **Σ(t)**: Fields, Keys, Kind, Constraints |
| Текст YASF (полный блок) | Локальная структурная реализация **R = ⟨G, Σ, C⟩** |

Тест считается пройденным, если компилятор порождает структурно идентичную локальную реализацию с точностью до порядка элементов в множествах. Набор conformance tests расширяется по мере появления новых конструкций YASF.

## F.10. Ограничения версии YASF

Не поддерживается: формальная грамматика (YASF определён через структуру секций и правила компиляции, а не через BNF/EBNF), автоматическая обработка свободного текста внутри секции «Обработка» (компилятор полагается на упомянутые имена таблиц и функций, а не на семантику свободного текста), наследование блоков (один блок YASF не может расширять другой), условные секции (ветвление внутри блока YASF не формализовано; `error-path` описывается через отдельную секцию «Ошибки»).

Запланировано: формальная грамматика YASF (для автоматической валидации синтаксиса), условные секции (для явного описания ветвлений внутри блока), поддержка шаблонов (типичные паттерны блоков, наследуемые и переопределяемые).
