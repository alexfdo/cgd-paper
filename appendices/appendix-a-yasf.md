# Appendix A — YASF specification (synchronized clean version)

## A.1. Purpose

YASF (Yet Another Specification Format) is the minimal semantic layer of the triple **T = ⟨Y, G, Σ⟩**. For small systems, it is a convenient authoring format in controlled natural language. For larger systems, YASF may also act as an **intermediate representation** generated by tooling. In both modes, YASF remains compilable and traceable.

## A.2. Design principles

1. **Controlled natural language.** Low entry threshold without introducing a new formal DSL as a mandatory interface.
2. **Deterministic compilation as an implementation invariant.** The same consistent YASF should map to the same graph and the same signatures.
3. **Minimality.** YASF records only the information required to construct Y, G, and Σ.
4. **Modularity.** The base block describes one function.
5. **Traceability.** Every graph element and signature must be traceable to a YASF section.

## A.3. Canonical subset of sections

For the purposes of the paper, the following canonical subset of sections is used:

- Purpose
- Input data
- Reference data
- Processing
- Output data
- Errors
- Side effects
- Triggers
- Contract

This subset is sufficient to construct the formal core of the paper, but it is not declared to be the only admissible YASF profile forever.

## A.4. Compilation rules YASF → G, Σ

1. Every function name in “Purpose” or “Triggers” creates a `function` node.
2. Every table name in the sections for inputs, reference data, outputs, errors, and side effects creates a `table` node.
3. Sections compile into edges: `consumes`, `reads`, `produces`, `writes`, `triggers`.
4. The six-component contract **C(f) = ⟨Cin, Cout, R, W, Tin, Tout⟩** is derived from the edges.
5. Table signatures **Σ(t)** are derived from table descriptions.
6. Any contract written explicitly by the author is checked against the derived one.

## A.5. Relation between YASF sections and the contract

- Input data → `Cin`
- Reference data → `R`
- Output data and Errors → `Cout`
- Side effects → `W`
- Triggers → `Tout`
- `Tin` is derived from other blocks that reference the current function in their “Triggers” section.

## A.6. Short example of a YASF block

**Function:** RegisterUser  
**Purpose:** Register a new user by email.  
**Input data:** RegistrationRequest.  
**Reference data:** User — for checking email uniqueness.  
**Processing:** accept the request, validate it, check uniqueness, create the user, emit the event, write the audit log, and trigger confirmation.  
**Output data:** User, RegistrationEvent.  
**Errors:** RegistrationError.  
**Side effects:** AuditLog.  
**Triggers:** SendConfirmation.  
**Contract:** `⟨{RegistrationRequest}, {User, RegistrationEvent, RegistrationError}, {User}, {AuditLog}, ∅, {SendConfirmation}⟩`.

## A.7. Decomposition in YASF

During decomposition, the aggregated block is preserved, while the subfunctions receive their own YASF blocks. The relation between levels is maintained through the `refinement_map` and by matching the external contract of the aggregated node with the contract of the subgraph after the trigger-internalization rule is applied.

## A.8. Verification-driven dialogue in YASF

After block compilation, defects are split into:

- **Blocking** — require an explicit user decision.
- **Auto-resolvable** — only safe normalizations: aliases, canonicalization of enum/PK/FK, kind, sections, and duplicate edges.
- **Advisory** — warnings and design recommendations.

Important: the absence of AuditLog on the error path is **not** considered auto-resolvable in the current version, because that decision belongs to the domain audit policy.

## A.9. Conformance tests

Conformance tests check pairs of the following form:

- YASF block → canonical subgraph;
- YASF block → canonical contract;
- table description → canonical signature.

## A.10. Version limitations

The current profile does not formalize:

- the full grammar of YASF;
- computable path-selection rules inside the Processing section;
- YASF block inheritance;
- per-consumer field contracts.
